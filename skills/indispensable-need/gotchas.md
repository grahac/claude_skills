# Indispensable Need Gotchas

Common failure patterns. Update this file every time something goes wrong.

## 1. Cheerleading instead of analyzing
**Problem:** Treating the analysis like a pitch deck — finding only positives, framing weaknesses as strengths.
**Fix:** The user wants truth, not validation. If the product is a commodity with no indispensable need, say so directly. Candor is the entire point.

## 2. Identifying a need that IS well-served by competitors
**Problem:** Finding an important job but ignoring that three competitors do it equally well. That's a commodity need, not an indispensable one.
**Fix:** Always pair importance with substitutability. An indispensable need requires BOTH high importance AND poor alternatives.

## 3. Being abstract instead of specific
**Problem:** "Users need a better workflow" or "People want to feel confident." These are true of every product ever built.
**Fix:** Name the specific user, the specific job, and the specific reason alternatives fail. If you can swap in a different product name and the statement still works, it's too vague.

## 4. Listing five needs instead of committing to one
**Problem:** Hedging by presenting multiple candidates without picking the strongest one.
**Fix:** Pick ONE. The skill demands conviction. Present the runner-ups briefly, but commit to the single most indispensable need and argue for it.

## 5. Skipping competitive research
**Problem:** Analyzing the product in isolation without actually searching for and evaluating alternatives.
**Fix:** WebSearch is not optional. Find real competitors, read real user complaints, understand the actual landscape. The substitutability analysis is meaningless without this.

## 6. Shallow codebase reading
**Problem:** Only reading the README or a few files and missing what the product actually does vs. what it claims to do.
**Fix:** When analyzing from codebase, read routes, schemas, core logic, and tests. The code reveals the real product — features that are half-built, features that are deeply invested in, and the actual user flows.

## 7. Confusing lock-in with indispensability
**Problem:** Claiming a product is indispensable because switching costs are high (data migration, integrations) when the actual need it serves is generic.
**Fix:** Lock-in is a moat, not a need. Distinguish between "can't leave because of switching costs" and "wouldn't leave because nothing else does this job." The latter is indispensability.

## 8. Agents producing redundant analysis
**Problem:** All four agents converge on the same insight from slightly different angles, wasting the multi-perspective approach.
**Fix:** Each agent has a distinct lens (functional, emotional, social, competitive). If they're all saying the same thing, the prompts weren't specific enough or the synthesis didn't extract unique value from each.

## 9. Ignoring the "not yet" case
**Problem:** Forcing an indispensable need onto a product that genuinely doesn't have one yet.
**Fix:** "Not yet indispensable" is a valid and useful finding. When this is the case, the most valuable output is what the product COULD build around to become indispensable.
